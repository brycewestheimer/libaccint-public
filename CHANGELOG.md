## [0.1.0-alpha.2] - 2026-03-03

### Summary

Second alpha release focused on making the published concurrency contract
defensible across CPU, CUDA, and MPI execution paths, while tightening release
packaging and developer validation workflows.

### Added

- Explicit alpha thread/process contract for CPU, GPU, and MPI usage
- Real CMake MPI backend detection and linkage
- `MPIFockBuilder` host-batch accumulation path for GPU-backed MPI execution
- In-tree Spack repository layout and recipe for local `spack repo add`
- Deterministic source-release builder at `packaging/release/build_source_release.sh`
- WSL2-safe CUDA+MPI test preset and updated developer runbook

### Changed

- GPU batched ERI device output now uses an owned handle-based API instead of an unsafe raw pointer path
- Raw-pointer GPU batch API now fails fast and remains only as an alpha-compatibility shim
- `GpuFockBuilder` accumulation now uses explicit stream ordering to avoid producer/consumer races
- `THREAD_SAFETY.md` is now the authoritative support matrix for concurrency guarantees
- Conda and Spack packaging track the deterministic release source asset instead of GitHub autogenerated tag archives

### Fixed

- Shared-engine CUDA concurrency lifetime hazard for device-resident ERI batches
- Missing GPU stream-dependency edges between ERI production and device-side Fock accumulation
- MPI build configuration not actually enabling the real MPI backend
- Stale thread-safety documentation around cached ShellSet worklist generation
- Non-blocking CUDA/MPI warning noise on the supported alpha build path

## [0.1.0-alpha.1] - 2026-02-24

### Summary

First public alpha release of LibAccInt. Exposes the stable public API
(ShellSets, ShellSetPairs/Quartets, Engine.compute/compute_and_consume, C++ and
Python APIs) with handwritten generic kernels only.

### Excluded from Alpha

- **Code generation framework** (`codegen/`) — reserved for publication
- **Generated kernels** (`generated/`, `src/generated/`) — reserved for publication
- **Full generated kernel registries** — stubs compiled unconditionally instead
- Internal development artifacts (`archive/`, `kernel_extension_development/`)

### Changed

- Default kernel dispatch strategy changed from `"optimal"` to `"handwritten"`
- `OptimalDispatchTable::populate_defaults()` now delegates to `populate_handwritten()`
- `cmake/CodeGeneration.cmake` returns early when `codegen/` directory is absent
- Build system unconditionally compiles registry stubs (no conditional on generated registry options)
- `test_build_pipeline.sh` rewritten as handwritten-only build smoke test

### Added — Phase 0.5: ShellSet Work Unit Integration

- **Canonical worklist accessors**: `BasisSet::shell_set_pairs()` and `shell_set_quartets()` with allocation-free caching after first call
- **Engine 1e ShellSetPair driver**: Full-basis one-electron computation (overlap, kinetic, nuclear) driven by ShellSetPair work units on both CPU and CUDA
- **Engine 2e ShellSetQuartet driver**: Full-basis two-electron compute-and-consume driven by ShellSetQuartet work units with GPU routing
- **Worklist overload**: `Engine::compute(op, span<const ShellSetQuartet>, consumer)` for custom scheduling, screening, and distributed execution
- **Example programs**: `examples/shellset_batched_1e.cpp` and `examples/shellset_batched_fock.cpp`
- **Migration guide**: `docs/user_guide/shellset-batching-migration.md` — before/after patterns for shell-level → ShellSet work units
- **Regression tests**: 11 new CPU tests + 5 Engine-level GPU routing tests (916 tests total)

### Changed

- **Engine orchestration**: `compute_1e_impl` and `compute_and_consume_impl` now iterate ShellSet work units instead of raw shell indices
- **CUDA 1e methods**: `compute_overlap_matrix`, `compute_kinetic_matrix`, `compute_nuclear_matrix` now iterate `shell_set_pairs()`
- **GPU dispatch**: `compute_and_consume` routes through ShellSetQuartet work units when GPU is selected
- **Design docs**: Updated `basis-set.md`, `engine.md`, `compute-and-consume.md` to match implemented APIs

### Deprecated

- `BasisSet::generate_shell_set_pairs()` — use `shell_set_pairs()` instead
- `BasisSet::generate_shell_set_quartets()` — use `shell_set_quartets()` instead

## [Planned v1.0.0]

- **CUDA GPU backend**: Full GPU acceleration via CUDA with batched kernels and device-side Fock accumulation
- **Unified compute() API**: Single entry point dispatching based on work unit type
- **True batched execution**: ShellSet operations execute in parallel (CPU) or single kernel launch (GPU)
- **Auto-tuning framework**: Analytical cost model with optional runtime profiling for optimal execution strategy
- **Intelligent CPU/GPU dispatch**: Automatic routing based on workload characteristics via DispatchPolicy
- **GPU Fock matrix builder**: Device-side J/K accumulation without host-device data transfers
- **Schwarz screening**: Integral prescreening for efficient ERI computation
- **Range-separated integrals**: Support for range-separated exchange functionals
- **First derivatives**: Gradient integrals for geometry optimization
- **Python bindings**: Full pybind11 bindings with NumPy integration
- **Code generation framework**: AM-specialized kernel generation via libaccint-codegen
- **Comprehensive documentation**: Sphinx docs with API reference, theory, and user guide
- **CI/CD pipeline**: Multi-platform GitHub Actions workflows (CPU, CUDA, HIP)
- **Docker images**: CPU, CUDA, and HIP container images
- **PyPI distribution**: Package available via pip install

### Changed

- **Engine architecture**: Restructured to support unified dispatch and multiple backends
- **ShellSet batching**: Groups shells by angular momentum for efficient GPU execution
- **Buffer management**: Improved memory layout with Structure-of-Arrays (SoA) for GPU coalescing
- **Threading model**: OpenMP-parallel ShellSetPair/ShellSetQuartet with per-thread buffers

## [0.1.0] - 2026-02-02

### Added

- **One-electron integrals**: Overlap (S), kinetic energy (T), and nuclear attraction (V) integrals using Obara-Saika recurrence relations and Rys quadrature.
- **Two-electron integrals**: Coulomb / electron repulsion integrals (ERI) via Rys quadrature with 2D VRR/HRR recursion over four centers.
- **FockBuilder consumer**: Coulomb (J) and exchange (K) matrix construction with fused compute-and-consume path to avoid materializing the ERI tensor.
- **Engine**: Central computation orchestrator with shell-pair and shell-quartet dispatch for one- and two-electron integrals.
- **Basis set infrastructure**: Shell, ShellSet, ShellSetPair, ShellSetQuartet, and BasisSet classes with Structure-of-Arrays (SoA) data layout.
- **Built-in STO-3G basis set**: Hard-coded data for H, C, N, O, F.
- **QCSchema JSON basis set parser**: Loads basis sets from Basis Set Exchange JSON format (STO-3G, cc-pVDZ, cc-pVTZ included).
- **Math utilities**: Boys function (asymptotic + Taylor series), Rys quadrature (tabulated roots/weights), Gaussian product theorem, Cartesian normalization.
- **CPU backend**: Host-side computation with OpenMP support.
- **Comprehensive test suite**: 648 tests covering all components, including integration tests validating against PySCF reference values.

### Supported Scope

- Angular momentum: s, p, d functions (Cartesian Gaussians)
- Basis sets: STO-3G (built-in), cc-pVDZ, cc-pVTZ (via JSON parser)
- Backend: CPU only
- Derivative order: energy only (DerivOrder=0)

### Quality Gate G2 Validation

- H2O/STO-3G RHF energy matches PySCF within 1e-10 Hartree
- H2/STO-3G and CH4/STO-3G energies validated against PySCF
- ERI 8-fold permutation symmetry verified
- Schwarz inequality holds for all shell quartets
- One-electron integrals match PySCF within 1e-12

### Known Limitations

- No integral screening (Schwarz or other)
- No density fitting / resolution of the identity
- No gradient or Hessian integrals
- No GPU backends (CUDA/HIP planned for future releases)
- No Python bindings
- Maximum angular momentum limited to d-functions
- No code-generated AM-specialized kernels
